ISSUE: Transcriber::new() immediately spawns a thread in the constructor

transcriber.rs:27-28: The constructor has the side effect of spawning a long-lived thread. This
means you cannot construct a Transcriber without immediately starting it. If the model file
doesn't exist, the error is reported asynchronously via the event channel rather than returned
from construction. This makes error handling at startup harder to reason about.

SOLUTION:

Split construction and startup into two phases:

    impl Transcriber {
        pub fn new(audio_rx: AudioReceiver, app_event_tx: AppEventSender, config: TranscriberConfig) -> Self {
            Self {
                audio_rx: Some(audio_rx),
                app_event_tx,
                config,
                handle: None,
            }
        }

        pub fn start(&mut self) -> Result<(), TranscriberError> {
            // Validate model exists before spawning thread
            if !std::path::Path::new(&self.config.model_path).exists() {
                return Err(TranscriberError::ModelNotFound(self.config.model_path.clone()));
            }
            let audio_rx = self.audio_rx.take().expect("start called once");
            let app_event_tx = self.app_event_tx.clone();
            let config = self.config.clone();
            self.handle = Some(thread::spawn(move || worker(audio_rx, app_event_tx, config)));
            Ok(())
        }
    }

This lets main.rs handle startup errors synchronously:

    let mut transcriber = Transcriber::new(audio_rx, app_event_tx.clone(), config.transcriber.clone());
    if let Err(err) = transcriber.start() {
        eprintln!("Transcriber failed to start: {err}");
        return;
    }

The Whisper model load itself (which is the expensive fallible operation) could also be moved
to start() if you want to catch it synchronously, though that would require loading the model
on the main thread and passing it into the worker.
