ISSUE: Recorder.stop() blocks the Controller's event loop via handle.join()

When the controller processes UiStopListening (controller.rs:109), it calls stop_listening()
which calls recorder.stop() (recorder.rs:203-215). This joins the recorder thread synchronously,
meaning the controller's 10ms event loop blocks until the recorder thread finishes. The recorder
thread sleeps in 20ms intervals checking the stop flag (recorder.rs:176-178), so in the worst
case this blocks for ~20ms. Any queued events wait during this time.

SOLUTION:

Make stop non-blocking by splitting it into two phases:

1. stop_signal() -- sets the stop flag and returns immediately
2. The recorder thread sends a Recorder::Stopped event when it finishes (it already does this
   at recorder.rs:192-195)
3. The Controller handles the Stopped event to clean up the join handle

    pub fn stop_signal(&self) {
        self.stop_flag.store(true, Ordering::SeqCst);
    }

In the Controller, on UiStopListening, call recorder.stop_signal(). On
Recorder::Stopped event, join the handle. This keeps the event loop responsive at all times.
The Recorder thread already sends the Stopped event, so the infrastructure is already in place.
