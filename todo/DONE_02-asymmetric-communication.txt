ISSUE: Asymmetric communication between UI and Controller

UI to Controller: proper channel-based events (app_event_tx.send(...))
Controller to UI: direct method calls on Ui struct + shared Arc<Mutex<UiState>>

This asymmetry means the UI is both an event producer (via the channel) and an event consumer
(via mutex polling). In most event-driven desktop apps, communication would be symmetric --
either both directions use channels, or both use shared state. The hybrid here makes it harder
to reason about data flow because you have to trace two different mechanisms.

SOLUTION:

Make communication symmetric by adding a dedicated Controller-to-UI channel. The Controller
sends typed messages (e.g., UiCommand::UpdateTranscription(String),
UiCommand::ProcessingComplete(String)) through this channel. The UI consumes them via an iced
Subscription. This pairs naturally with fixing issue #1 -- both fixes converge on the same
approach: a channel-backed iced subscription for Controller-to-UI communication, mirroring
the existing channel-based UI-to-Controller path.
