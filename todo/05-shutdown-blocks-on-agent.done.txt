ISSUE: Shutdown blocks indefinitely waiting for in-flight agent request

When the controller shuts down, it drops the Agent, whose Drop impl
joins the worker thread. If the agent is mid-request to OpenAI, the
join blocks until that request completes or times out (up to 60s per
REQUEST_TIMEOUT_SECS, potentially longer with retries — up to 5
attempts with exponential backoff totaling ~90s worst case).

During this time the application window has already closed (UI exited
first), so the user sees a hung process with no visible window.

FILES: src/agent.rs (Drop impl), src/controller.rs (run() cleanup)

FIX: Signal the worker thread to stop before joining. Add a stop flag
(AtomicBool) that the worker checks between retries, and set it before
dropping the sender. This lets the worker abort a retry loop promptly.

  pub struct Agent {
      tx: AgentSender,
      stop: Arc<AtomicBool>,
      handle: Option<JoinHandle<()>>,
  }

In the worker, check `stop` between retries in call_openai_with_retry
and after each recv():

  while let Ok(request) = rx.recv() {
      if stop.load(Ordering::SeqCst) { break; }
      // ... process request, checking stop between retries
  }

In Drop:

  fn drop(&mut self) {
      self.stop.store(true, Ordering::SeqCst);
      // Drop sender to unblock recv()
      drop(std::mem::take(&mut self.tx));  // or restructure to drop explicitly
      if let Some(handle) = self.handle.take() {
          let _ = handle.join();
      }
  }

For the in-flight HTTP request itself, consider using a shorter timeout
or a cancellation-aware HTTP client (though reqwest blocking doesn't
support cancellation — the stop flag at least prevents starting new
retries).
