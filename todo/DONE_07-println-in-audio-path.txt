ISSUE: println! in audio callbacks and transcriber hot path

recorder.rs:96, recorder.rs:115, recorder.rs:137 use println! in cpal audio stream callbacks.
transcriber.rs:88 uses println! in the transcription result path.

In a hot audio callback path, stdout writes can block on a full pipe buffer and introduce audio
glitches. The recorder callbacks (println!("Got f32 chunk of size {}", ...)) are especially
problematic since they run on cpal's real-time audio thread where any blocking is unacceptable.
These appear to be leftover debug prints.

SOLUTION:

Remove all println! calls from audio callbacks and the transcriber:

- recorder.rs:96  -- delete: println!("Got f32 chunk of size {}", chunk.len());
- recorder.rs:115 -- delete: println!("Got i16 chunk of size {}", chunk.len());
- recorder.rs:137 -- delete: println!("Got u16 chunk of size {}", chunk.len());
- transcriber.rs:88 -- delete: println!("Transcribed: {}", text);

The log::debug! calls already present nearby provide the same information through the logging
system, which writes to a file asynchronously and won't block the audio thread. No replacement
is needed.
