ISSUE: Controller pushes state to UI by directly calling methods, bypassing iced's MVU mechanism

The Controller holds a Ui clone and calls ui.refresh_with_state(snapshot) (controller.rs:146)
and ui.submit_processed_text(text) (controller.rs:66) from its background thread. These methods
mutate Arc<Mutex<UiState>> behind iced's back. The iced framework then discovers these changes
during its Tick subscription (ui.rs:525-541) by checking repaint_requested.

This is essentially polling for side effects -- iced's MVU loop has no idea why the state changed,
it just notices a flag was flipped. In a conventional iced app, external data arrives through
subscriptions (e.g., an async channel subscription that yields Message variants).

Consequences:
- The UI can be up to 16ms stale even when data is available immediately
- The editor content is reconstructed from scratch on every repaint (ui.rs:530:
  text_editor::Content::with_text(...)) instead of being updated incrementally, which destroys
  cursor position and selection state
- There's a subtle race: the Controller and iced's update function both access UiState through
  the same mutex

SOLUTION:

Replace the mutex-based push with an iced Subscription that reads from a channel. Create an
mpsc or tokio channel from Controller to UI. In the iced subscription function, yield Message
variants (e.g., Message::TranscriptionUpdated(String), Message::AgentResponseReceived(String))
when data arrives on the channel. This lets iced drive state changes through its own update()
function, eliminates the shared UiState mutex for cross-thread communication, and removes the
polling delay. The editor content can then be updated incrementally in response to specific
messages rather than being reconstructed wholesale.
