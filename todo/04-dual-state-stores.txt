ISSUE: Two separate state stores that must be kept in sync

AppState (shared between threads) and UiState (owned by the UI) both contain the text being
edited. The input field in UiState and transcribed_text in AppState are synchronized through:
- sync_text_to_app_state() -- pushes UI edits to AppState before certain operations
- refresh_with_state() -- pulls AppState into UiState when listening

This dual-state design requires careful coordination. If you forget to call
sync_text_to_app_state() before an operation that reads from AppState, you get stale text.
The submit() method does remember (ui.rs:419), but this is convention, not enforcement.

SOLUTION:

Designate a single source of truth for the editor text. Two approaches:

Option A: AppState is the source of truth. The UI editor is a view of AppState. Every
keystroke sends a debounced UiUpdateText event to the Controller, which updates AppState.
The UI always renders from the latest snapshot. This is simpler but may feel laggy for typing.

Option B: UiState is the source of truth during editing. AppState only stores text when the
user is not actively editing (i.e., during listening or after submit). The Controller reads
text from the UiSubmitText event payload rather than from AppState. This eliminates the need
for sync_text_to_app_state() entirely -- the submit event already carries the text.

Option B is the lower-friction change. The key insight is that the Controller already receives
the text in UiSubmitText(text) (controller.rs:111), so it never needs to read it from AppState.
