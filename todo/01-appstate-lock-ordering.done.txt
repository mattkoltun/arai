ISSUE: AppState uses separate mutexes with inconsistent lock ordering

update_prompts() locks: agent_prompts -> default_prompt -> config
agent_instruction() locks: agent_prompts -> default_prompt

But snapshot() locks them independently, and there is a window between
updating agent_prompts and default_prompt where a concurrent reader can
see new prompts with the old default index (or vice versa). Since
default_prompt is an index into agent_prompts, this can produce an
out-of-bounds read that silently returns an empty instruction.

More importantly, using four separate mutexes for fields that must be
consistent with each other invites this class of bug every time a new
reader or writer is added.

FILES: src/app_state.rs

FIX: Replace the four independent Mutex<T> fields with a single
Mutex<AppStateInner> (or RwLock<AppStateInner>) that holds all fields
together. Every read or write acquires one lock, guaranteeing a
consistent view.

  struct AppStateInner {
      agent_prompts: Vec<AgentPrompt>,
      default_prompt: usize,
      transcriber: TranscriberConfig,
      config: Config,
  }

  pub struct AppState {
      inner: Mutex<AppStateInner>,
  }

snapshot(), agent_instruction(), update_prompts(), and
update_transcriber() each lock once and operate on a consistent state.
