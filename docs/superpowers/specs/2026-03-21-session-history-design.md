# Session History

## Overview

Every time the user triggers copy+hide (Cmd+Enter or Cmd+C), the copied text is saved to a JSON file in `~/.local/share/arai/history/`. Each file represents one session. A background thread handles file I/O to keep the UI and controller responsive.

## File Storage

### Directory

`~/.local/share/arai/history/`

Created on first write if it doesn't exist.

### File Naming

Incremental zero-padded IDs: `0001.json`, `0002.json`, etc. Zero-padded to 4 digits; after 9999, continues with 5+ digits (no overflow limit).

The worker thread keeps a counter starting from `max_existing + 1` (determined at startup by scanning the directory). The counter only increments â€” never reuses IDs.

### ID Scan

On startup, the worker thread scans the history directory:
1. `std::fs::read_dir()` to list entries
2. Filter filenames matching `*.json`
3. Parse the numeric prefix (strip `.json`, parse as `u64`)
4. Find the maximum
5. Set next ID to `max + 1` (or 1 if no files exist)

### File Lock

No external crate. Use `std::fs::File::create()` on `~/.local/share/arai/history/.lock` combined with `fs2::FileExt::lock_exclusive()` â€” actually, to avoid adding a dependency, use a simpler approach: since the worker thread is single-threaded and the counter is local to the thread, concurrent instances are unlikely. The ID scan + write happens atomically within the worker thread. If a collision occurs (file already exists), log a warning and increment the ID.

### File Format

```json
{
  "text": "the copied text",
  "timestamp": "2026-03-21T14:32:05Z",
  "prompt": "Rewrite clearly"
}
```

- `text`: The text that was copied to clipboard.
- `timestamp`: ISO 8601 UTC timestamp, generated by the worker thread at write time using manual formatting from `SystemTime` (same approach as `format_timestamp()` in controller.rs, but ISO 8601 format).
- `prompt`: The name of the active agent prompt at copy time.

## Architecture

### New Module: `src/history.rs`

A background worker thread with an mpsc channel, following the same pattern as `Recorder` and `Transcriber`.

```rust
pub struct History {
    tx: std::sync::mpsc::Sender<HistoryEntry>,
    handle: Option<std::thread::JoinHandle<()>>,
}

struct HistoryEntry {
    text: String,
    prompt: String,
}
```

**`History::new()`** â€” spawns the worker thread that:
1. Creates the history directory if needed
2. Scans for existing files to determine next ID
3. Loops on `rx.recv()`
4. For each entry: generates timestamp, writes JSON file with current ID, increments ID
5. If file already exists (collision), increments ID and retries

**`History::save(text, prompt)`** â€” sends a `HistoryEntry` to the worker thread via the channel. Non-blocking from the caller's perspective.

**`Drop` for `History`** â€” drops the sender (causing the worker to exit on next recv) and joins the thread. Any entries buffered in the channel but not yet processed will be lost. This is acceptable since history is best-effort.

### New Event: `AppEventKind::UiCopied`

```rust
AppEventKind::UiCopied { text: String, prompt: String }
```

Sent from the UI to the controller when the copy+hide action fires.

### Data Flow

```
UI (Message::Copy) â†’ sends UiCopied { text, prompt } â†’ Controller â†’ history.save(text, prompt) â†’ History thread â†’ writes file
```

- Add `UiCopied { text: String, prompt: String }` variant to `AppEventKind`

### `src/ui.rs`

- Create `History` instance in `Controller::new()` or `run()`
- Handle `(AppEventSource::Ui, AppEventKind::UiCopied { text, prompt })` â€” call `self.history.save(text, prompt)`

### `src/history.rs` (new)
- `History` struct with channel + thread handle
- Worker thread: directory creation, ID scan, JSON write
- `save()` method
- `Drop` implementation
- No external dependencies â€” uses `serde`/`serde_json` (already in Cargo.toml) and `std::time::SystemTime` for timestamps

### `src/main.rs`
- Add `mod history;`

## Error Handling

- Directory creation failure: logged, save skipped
- File write failure: logged, save skipped
- Channel send failure: logged (History was dropped)
- No panics â€” history is best-effort, never blocks the app
